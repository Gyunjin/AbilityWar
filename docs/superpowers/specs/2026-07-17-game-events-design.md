# AbilityWar 게임성 강화 설계 — 이벤트 시스템과 정보 공개

작성일: 2026-07-17

## 배경

3~6명이 모여 하룻밤에 2~3판 하는 소규모 사설 서버. 한 판 목표 길이는 20~30분.

현재 구조의 공백:

- **파밍 구간이 비어 있다.** 능력 6종이 전부 전투용이라 파밍 중엔 쓸 일이 없고, 자기장 외에는 아무 사건도 없다.
- **정보가 없다.** 누가 어떤 능력인지 게임이 끝나야 공개된다. 게임 중 심리전이 성립하지 않는다.
- **`killCounts`가 죽어 있다.** 집계만 하고 게임에 영향을 주지 않는다.
- **매 판이 똑같다.** 반복 플레이 가치가 낮다.

## 목표

1. 파밍 구간에 판단거리를 준다.
2. 매 판 다른 전개를 만든다.
3. 능력 정보를 단계적으로 공개해 심리전을 만든다.

비목표(이번 범위 밖):

- 킬 보상/성장 시스템 — 스노우볼을 유발해 판이 일찍 끝난다.
- 새 능력 추가 — 3~6명에 6종이면 충분하다.
- ChatColor → Adventure 마이그레이션(B그룹) — 별도 작업.

## 1. 아키텍처

이벤트 추가는 능력 추가와 동일하게 만든다: 클래스 하나 + 목록에 한 줄. `Main`은 손대지 않는다.

### GameEvent 인터페이스

```java
public interface GameEvent {
    String getName();                                          // "보급 투하"
    void start(GameContext ctx);                               // 발동
    default boolean canRun(GameContext ctx) { return true; }   // 발동 가능 조건
}
```

**`stop()`도 `durationSeconds()`도 두지 않는다.** 5개 이벤트를 하나씩 확인한 결과 전부 순간 이벤트다:

- 전원 발광 / 현상수배 — `addPotionEffect(GLOWING, 30초/60초)`로 **스스로 만료**된다. 티모 영구 투명 버그는 `Integer.MAX_VALUE`를 걸어서 났던 것이지, 유한 지속시간은 갱신도 해제도 필요 없다.
- 사령의 밤 — 밤은 자연히 지나간다.
- 보급 투하 / 능력 재충전 — 발동 즉시 끝.

지속 추적이 필요 없으므로 `GameEventManager`에서 만료 스케줄링이 통째로 사라진다. 스폰물 정리는 게임 종료 시 PDC 스윕 한 번으로 처리한다(5절).

향후 지속형 이벤트가 실제로 필요해지면 그때 훅을 추가한다. 지금 미리 만들지 않는다.

### GameContext

이벤트가 필요로 하는 것만 담는다. **이벤트는 `Main`을 직접 참조하지 않는다.** 그래야 이벤트를 독립적으로 이해하고 교체할 수 있다.

- 게임 월드
- 생존자 목록
- 파밍 구간 여부 (`canRun`이 전투 전용 이벤트를 거르는 데 사용)
- 킬 집계 조회 (현상수배용)

경과 시간은 담지 않는다 — 5개 이벤트 중 쓰는 것이 없다.

### GameEventManager

- `Main.gameTimer`에서 1초마다 `tick()`을 받는다. 새 스케줄러를 만들지 않는다(능력의 `checkPassiveAbilities`와 동일한 방식).
- `이벤트간격(초)`마다 추첨 → `canRun()`이 true인 후보 중 무작위 선택 → `start()`.
- **직전에 나온 이벤트는 다음 추첨에서 제외**한다(연속 중복 방지).

**별도의 레지스트리 클래스를 만들지 않는다.** `AbilityRegistry`에 `create(name)`/`isValid()`/`getNames()`가 있는 것은 `/능력변경` 명령어와 탭 완성이 그것들을 쓰기 때문이다. 이벤트에는 그런 명령어가 없으므로 같은 API를 복사하면 아무도 호출하지 않는 코드만 늘어난다. `GameEventManager` 안의 목록 하나로 충분하며, 추가 방법은 능력과 똑같이 한 줄이다:

```java
private static final List<Supplier<GameEvent>> EVENTS = List.of(
        SupplyDropEvent::new,
        NightRushEvent::new,
        AbilityRechargeEvent::new,
        GlowingEvent::new,
        BountyEvent::new);
```

## 2. 이벤트 목록

구간 전용을 최소화해, 이벤트 수를 늘리지 않고 조합을 늘린다.

### 공용 (파밍·전투 모두)

**보급 투하**
- **현재 자기장 크기 안쪽** 무작위 좌표에 상자 스폰, 좌표를 전체 공지.
  - 자기장은 (0,0) 중심이므로 `getWorldBorder().getSize() / 2`를 반경으로 쓴다. 축소가 진행 중이면 그 시점의 실제 크기를 쓰므로, 후반에는 상자가 중앙 근처에만 뜬다.
  - 지면 높이는 `getHighestBlockYAt()`으로 잡고, 청크를 먼저 강제 로드한다(로드 전에 호출하면 부정확한 높이가 나온다 — `getGameSpawnLocation()`에 같은 처리가 있다).
- 내용물: 장비/황금사과/음식.
- **상자 주변에 몬스터 3~4마리를 함께 스폰한다.**
- 근거: 파밍 중엔 PVP가 안 되므로 상자만 놓으면 가장 가까운 사람이 걸어가 먹고 끝난다. 경쟁이 성립하지 않는다. 몬스터를 두면 "뚫을 수 있나"라는 판단이 생기고, 파밍 무적을 푼 변경이 여기서 값을 한다.

**사령의 밤**
- 시간을 밤으로 돌리고, 각 생존자 주변에 몬스터를 스폰한다. 발동 시 1회로 끝난다.
- 밤은 자연히 지나가므로 시간을 되돌리지 않는다. 스폰된 몬스터는 PDC 표식을 달아 게임 종료 시 회수한다.

**능력 재충전**
- **전원의 능력 쿨타임을 즉시 초기화**하고 전체 공지한다.
- 지속 시간 없음(순간 이벤트).
- 근거: 이 게임에만 있을 수 있는 유일한 이벤트. 헐크 20초·블링커 8초·네크로 35초·사망회귀 **5분**이 한순간에 준비 완료된다. 특히 사망회귀 5분 초기화는 판을 뒤집는 사건이고, 3절의 능력 공개와 맞물려 "이번 판에 사망회귀가 있다 → 방금 재충전됐다 → 방심하면 안 된다"는 계산을 만든다.
- **"30초간 무제한"이 아니라 "즉시 초기화"로 잡은 이유**: 지속 시간 추적도 종료 시 원복도 필요 없는 순간 이벤트라, 정리 누락 버그가 원천적으로 생기지 않는다.
- **파밍 중에도 유효한가?** 유효하다. 파밍 중엔 PVP가 잠겨 공격 능력은 막히지만, 블링커 대쉬(이동)·네크로맨서 소환(PVE)·티모 은신은 그대로 쓰이므로 쿨타임이 돈다. 다만 전투 구간보다 효과가 약한 것은 사실이며, 이는 의도적으로 감수한다 — "아무도 쿨타임을 안 쓴 상태"를 감지해 건너뛰려면 `Ability.isOnCooldown()` 훅이 추가로 필요한데, 파밍 이벤트 풀이 3개뿐이라 건너뛴 뒤 뽑을 대안도 마땅치 않다. 실플레이 후 파밍에서 자주 헛돈다고 판단되면 그때 전투 전용으로 옮긴다.

`Ability` 인터페이스에 훅을 하나 추가한다:

```java
/** 능력 재충전 이벤트 등에서 쿨타임을 즉시 초기화할 때 호출됩니다. */
default void resetCooldown() {}
```

각 능력은 `lastUsed = 0;` 한 줄. 포세이돈은 쿨타임이 없어 기본 no-op을 쓴다.

### 전투 전용

**전원 발광**
- 30초간 전 생존자에게 GLOWING.
- 근거: 자기장은 물리적으로 좁히고, 발광은 정보로 좁힌다. 최후반 10x10 대치를 강제로 깨는 장치.

**현상수배**
- 최다 킬러에게 60초 GLOWING + 전체 공지.
- **전원 0킬이면 `canRun()`이 false를 반환해 다른 이벤트를 다시 추첨한다.**
- 근거: 보상이 아니라 **페널티**다. 앞서가는 사람의 위치를 노출해 나머지가 견제하게 만드는 **안티 스노우볼 장치**. 이미 집계 중인 `killCounts`를 처음으로 게임에 쓴다.

### 채택하지 않은 것

- **유성우, 저중력** — 바닐라 물리를 비트는 아케이드 요소라 "마인크래프트 + 능력자"라는 게임 픽션에서 벗어난다.

### 조합 수

파밍 2회 추첨(공용 3개 중) × 전투 2회 추첨(전체 5개 중) = 한 판 **약 30가지 조합**. 이벤트 5개로 하룻밤 2~3판이 겹치지 않는다.

## 3. 정보와 심리전

정보를 한 번에 다 주지도, 끝까지 감추지도 않고 단계적으로 푼다.

| 시점 | 공개 내용 |
|---|---|
| 파밍 중 | 아무것도 공개하지 않음 (현행 유지) |
| **파밍 종료 = PVP 시작** | **이번 판에 등장한 능력 이름 목록.** 누구 것인지는 비공개 |
| **플레이어 사망 시마다** | **그 사람의 능력** 공개 |
| 게임 종료 | 전원 능력 + 킬수 (현행 유지) |

가운데 두 개가 신규다. 이 둘이 맞물려 **소거법**이 성립한다. 5명이 시작하고 능력 5종이 공지되면, 사망자의 능력이 밝혀질 때마다 남은 사람의 후보가 좁혀진다. 후반으로 갈수록 정보가 선명해지고 그것이 최후반 교전의 긴장을 만든다.

구현은 기존 코드 재활용:
- 등장 목록 공개: `Main.revealAllAbilities()`를 참고해 이름만 섞어 출력.
- 사망 공개: `Main.onPlayerDeath()`에 브로드캐스트 추가.

### 능력 중복 없이 배정 (선행 조건)

현재 `AbilityManager.assignAbilitiesIfNone()`은 `AbilityRegistry.randomName()`을 플레이어마다 독립적으로 호출해 **같은 능력이 중복 배정될 수 있다.**

중복이 있으면 위 설계가 무너진다:
- "등장 능력 목록"의 정보량이 줄고, 목록 길이로 중복 여부를 역산할 수 있다.
- **소거법이 성립하지 않는다.** "티모가 죽었다"가 나와도 티모가 또 있을 수 있다.

**변경**: 능력 목록을 섞어 한 명씩 배정한다. 인원이 능력 수(6)를 초과할 때만 그때부터 중복을 허용한다. 3~6명이면 항상 전원 다른 능력이 된다.

## 4. 설정

기존 `/게임설정` 패턴을 따라 항목 하나만 추가한다.

| 설정명 | 기본값 | 비고 |
|---|---|---|
| `이벤트간격(초)` | 300 | 0이면 이벤트 시스템 비활성화 |

config 키는 `cfg-event-interval`. 탭 완성 목록에도 추가한다.

**파밍/전투 시간의 코드 기본값은 건드리지 않는다.** 목표인 20분 한 판(파밍 12분 + 전투 8분)은 이미 있는 명령어로 설정하면 되고, config에 저장되어 재시작해도 유지된다:

```
/게임설정 평화시간(초) 720
/게임설정 전투시간(초) 480
```

기본값 상수를 바꾸는 것은 이 설계에 필요 없는 변경이다.

`이벤트간격(초)`을 설정 가능하게 두는 이유는 두 가지다. 실플레이 후 리듬을 조정해야 하고(5분이 최선인지는 돌려봐야 안다), **검증 시 30초로 줄여 빠르게 확인**해야 한다.

## 5. 정리 책임 (Error Handling)

이 프로젝트에서 실제로 발생했던 사고들을 반복하지 않는 것이 목표다.

- **스폰물(몬스터·상자)은 전부 PDC 표식을 단다.** 네크로맨서 좀비가 `FixedMetadataValue`(런타임 전용, 저장 안 됨)를 써서 재시작 후 영구히 제거 불가능해진 사고가 있었다. PDC는 엔티티와 함께 저장된다.
- **정리는 게임 종료 시 스윕 한 번으로 한다.** `stopGameForce()`와 `onDisable()`에서 PDC 표식이 붙은 스폰물을 회수하고, 보급 상자는 블록째 제거한다. 이벤트별 `stop()`이 없으므로 정리 지점이 한 곳뿐이고, 그래서 빠뜨릴 곳도 한 곳뿐이다.
- **포션 효과는 유한 지속시간으로만 건다.** 무한 지속시간을 한 번 걸고 해제에 실패하면 영구히 남는다(티모 영구 투명 버그의 원인). 30초/60초짜리는 스스로 만료되므로 해제 코드 자체가 필요 없다.
- 이벤트 하나가 예외를 던져도 게임 타이머가 죽지 않도록 `start()` 호출을 try-catch로 감싸고 로그를 남긴다.

## 6. 검증

**이 프로젝트는 테스트가 0개이고, 서버 없이는 자동 검증이 불가능하다.** 컴파일 성공은 동작을 보장하지 않는다(실제로 "컴파일은 되는데 텔레포트가 전부 실패"하던 버그가 있었다).

따라서 이벤트마다 수동 검증 체크리스트를 둔다. `이벤트간격(초) 30`으로 설정해 빠르게 확인한다.

| 항목 | 확인 방법 |
|---|---|
| 이벤트 추첨 | 30초 간격으로 공지가 뜨는가. 같은 이벤트가 연속으로 나오지 않는가 |
| 구간 제한 | 파밍 중 전원 발광/현상수배가 뽑히지 않는가 |
| 보급 투하 | 공지된 좌표에 상자와 몬스터가 실제로 있는가 |
| 사령의 밤 | 밤이 되고 몬스터가 스폰되는가 |
| 능력 재충전 | 쿨타임 중이던 능력이 즉시 사용 가능해지는가 (사망회귀 포함) |
| 전원 발광 | 벽 너머로 윤곽이 보이는가. 30초 후 사라지는가 |
| 현상수배 | 전원 0킬일 때 건너뛰는가. 최다 킬러가 정확한가 |
| 능력 공개 | 파밍 종료 시 목록이 뜨는가. 사망 시 그 사람 능력이 공개되는가 |
| 중복 없는 배정 | 5명 게임에서 전원 다른 능력인가 |
| 정리 | `/게임종료` 후 스폰된 몬스터/상자가 사라지는가. 발광이 풀리는가 |

## 구현 순서 (제안)

1. `GameEvent` / `GameContext` / `GameEventManager` 뼈대 + `Main` 연결
2. 능력 중복 없이 배정 (독립적, 즉시 효과)
3. 능력 공개 2종 (독립적, 즉시 효과)
4. 이벤트 3종 (보급 투하 · 사령의 밤 · 능력 재충전) + `Ability.resetCooldown()` 훅
